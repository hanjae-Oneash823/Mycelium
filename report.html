what it is
A personal productivity suite wrapped in a retro BIOS/terminal interface. The app is a plugin host — a fullscreen black canvas that routes between self-contained tools via a navigation menu and a radial "bloom" button always visible in the corner.
current state
The core shell (plugin host, nav, animations) is solid. The Notes plugin is production-quality. Geo-Portal is visually complete. Everything else in the menu is a stub. The database schema is forward-looking but mostly unused.

architecture
startup flow · plugin system · state management
startup flow
index.html
└─ splash screen // animated dot, 2s minimum visible └─ main.jsx // React mount, fades splash after 2s └─ App.jsx // calls setupDb() on mount └─ lib/db.js // connects SQLite, runs CREATE TABLE IF NOT EXISTS for all tables └─ shows "Preparing o_neash..." until DB ready └─ renders AppContent ├─ AlwaysOnTop // z-index:1000 · home btn + bloom nav └─ PluginBox // renders active plugin or HomePage
plugin system
Three files form the entire plugin infrastructure:
  • registry.js — static array of {id, name, component}
  • PluginBox.jsx — looks up active plugin, renders with Framer Motion
  • SingleBloomNav.jsx — reads registry to generate nav petals
Adding a plugin = one line in registry.js. No dynamic imports, no lazy loading, no manifests.
state management
Zustand store (usePluginStore) holds two pieces of state:
  • plugins — array loaded from registry.js
  • activePlugin — string ID or null (home)
setActivePlugin(id) is the only action. LaunchMenu, SingleBloomNav, and the home button all consume the store independently.
plugin switching flow
user clicks LaunchMenu item or SingleBloomNav petal
→ setActivePlugin("notes" | "geo-portal" | "todo-list" | null) → PluginBox reads activePlugin from Zustand store → Framer Motion AnimatePresence (mode="wait") → old component fades out y:-10, opacity:0 — 300ms → new component fades in y:10→0, opacity:0→1 — 300ms → if null → renders HomePage

notes plugin built
The most complete plugin. Full note management with groups, tagging, and a custom force-directed graph visualization.
  • groups: default + user-created, persisted as JSON via FS
  • two-column layout: graph/list left · editor right
  • custom canvas force-directed graph — no D3
  • physics sim: 300 iterations, smooth cubic ease camera pan
  • note nodes (white) + tag nodes (blue), hover glow, search dim
  • CRUD: notes + groups with confirmation dialogs
  • #tag support with search filtering by title, content, tags
geo-portal built
3-page state machine with retro terminal landing, BIOS-style loading sequence, and a fully functional interactive map.
  • landing: terminal text + spinning earthspin.gif

  • map: MapLibre GL 5 + custom PMTiles raster tiles
  • zoom 3–10, navigation controls
  • 73% map / 27% side panel (placeholder)
  • CSS fade-in/out between all 3 pages

filesystem vs sqlite — notes
The Notes plugin uses tauri-plugin-fs to read/write JSON files at notes/{groupId}/{noteId}.json. This was intentional for fast iteration. The SQLite notes tables (groups, notes, tags, junc_note_tags) are designed and ready for a future migration if querying or cross-plugin access is needed.

aesthetic
retro terminal / BIOS / pixel art. Every decision reinforces this. Pure black canvas, monospace fonts, uppercase labels, lowercase buttons, sharp corners everywhere (border-radius:0), thin 1px borders, letter-spacing 2–6px. No scrollbars. Full-viewport layout. The UI feels like a custom OS shell, not a web app.
