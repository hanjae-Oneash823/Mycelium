# O-neash Database Schema

**File:** `~/Documents/O-neash-data/oneash-DB.db`
**Engine:** SQLite (via `@tauri-apps/plugin-sql`)
**Schema applied:** Every app launch via `setupDb()` in `src/lib/db.ts`

The schema uses `CREATE TABLE IF NOT EXISTS`, so it is safe to run on every startup â€” it only creates tables that don't yet exist. Column additions happen via try/catch `ALTER TABLE ADD COLUMN` migrations at the bottom of `setupDb()`.

---

## Tables Overview

| Table | Purpose |
|---|---|
| `arcs` | Top-level goal containers (like life chapters or quarterly themes) |
| `projects` | Sub-groupings within an arc |
| `planner_groups` | User-defined color-coded tags applied to nodes |
| `nodes` | Every task and event in the planner |
| `node_groups` | Junction table â€” which groups a node belongs to |
| `sub_tasks` | Checklist items nested inside a node |
| `productivity_logs` | One record per node completion â€” used for tracking output over time |
| `user_capacity` | Single-row user settings for daily work capacity and peak hours |
| `tendril_edges` | Directed links between nodes in the Tendrils view |
| `routines` | Recurring task/event templates |
| `routine_rules` | The specific recurrence schedule rules for a routine |
| `routine_groups` | Junction table â€” which groups a routine belongs to |

## `arcs`

An **arc** is the highest level of organisation. Think of it as a life chapter, a quarter, a theme, or a major goal. Arcs contain projects, and projects contain nodes.

```sql
CREATE TABLE IF NOT EXISTS arcs (
  id         TEXT PRIMARY KEY,
  name       TEXT NOT NULL,
  color_hex  TEXT DEFAULT '#00c4a7',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

| Column | Type | Description |
|---|---|---|
| `id` | TEXT | UUID generated by `crypto.randomUUID()` at insert time |
| `name` | TEXT | Display name (e.g. "Q1 2026", "Learning Track") |
| `color_hex` | TEXT | Hex colour used throughout the UI to tint the arc and its children. Defaults to teal `#00c4a7` |
| `created_at` | TIMESTAMP | Auto-set on insert, used for ordering (arcs load oldest-first) |

**App usage:**
- Created via `ArcForm.tsx` â†’ `createArc()` in `plannerDb.ts`
- Loaded once on startup into `usePlannerStore` and kept in memory
- Colour cascades to all child nodes that don't have a more specific override
- Displayed in the arc selector dropdowns in `TaskForm`, `RoutineForm`, and `ProjectFields`
- Visible in `TodayView` as a summary widget ("active arcs" count)
- Visible in `TendrilsView` as grouped project clusters
- Deleting an arc sets `arc_id = NULL` on all its nodes and projects (FK `ON DELETE SET NULL`)

---

## `projects`

A **project** is a focused body of work inside an arc. Nodes are optionally linked to a project for grouping and visual colour-coding.

```sql
CREATE TABLE IF NOT EXISTS projects (
  id         TEXT PRIMARY KEY,
  arc_id     TEXT,
  name       TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY(arc_id) REFERENCES arcs(id) ON DELETE SET NULL
);
```

| Column | Type | Description |
|---|---|---|
| `id` | TEXT | UUID |
| `arc_id` | TEXT | FK to `arcs.id`. NULL if the project is not attached to an arc |
| `name` | TEXT | Display name (e.g. "Website Redesign", "Tax Filing") |
| `created_at` | TIMESTAMP | Auto-set on insert |

**App usage:**
- Created via `ProjectForm.tsx` â†’ `createProject()` in `plannerDb.ts`
- Loaded into `usePlannerStore` alongside arcs on startup
- Project dropdowns in task/routine forms are filtered by the selected `arc_id`
- When a node has both `arc_id` and `project_id`, the arc's colour is used for dot tinting
- Deleting a project orphans its nodes (`project_id = NULL`) but does not delete them
- Deleting an arc sets `arc_id = NULL` on its projects

## `planner_groups`

**Groups** are user-defined colour labels that can be applied to any node. They work like tags â€” one node can belong to multiple groups. There is always one system group called **ungrouped** that is automatically applied to any node that has no real group.

```sql
CREATE TABLE IF NOT EXISTS planner_groups (
  id           TEXT PRIMARY KEY,
  name         TEXT NOT NULL,
  color_hex    TEXT DEFAULT '#64c8ff',
  sort_order   INTEGER DEFAULT 0,
  is_visible   BOOLEAN DEFAULT 1,
  is_ungrouped BOOLEAN DEFAULT 0,
  created_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

| Column | Type | Description |
|---|---|---|
| `id` | TEXT | UUID. The system ungrouped group always has the fixed id `g-ungrouped` |
| `name` | TEXT | Display name (e.g. "Deep Work", "Health", "Admin") |
| `color_hex` | TEXT | Hex colour used for dot tinting and badges. Defaults to blue `#64c8ff` |
| `sort_order` | INTEGER | Controls the order groups appear in the sidebar filter. Lower = higher |
| `is_visible` | BOOLEAN | Whether the group is shown in the sidebar filter UI |
| `is_ungrouped` | BOOLEAN | `1` only for the `g-ungrouped` system group. Identifies the fallback group |
| `created_at` | TIMESTAMP | Auto-set on insert |

**App usage:**
- The `g-ungrouped` row is seeded at startup via `INSERT OR IGNORE` â€” it always exists
- Created via the group management UI â†’ `createGroup()` in `plannerDb.ts`
- Loaded into `usePlannerStore` and displayed as filter pills in the sidebar
- When a node is inserted with no `group_ids`, the `nodes_auto_ungrouped` trigger automatically adds `g-ungrouped` to `node_groups`
- When all real groups are removed from a node, the `readd_ungrouped_if_empty` trigger restores the `g-ungrouped` membership
- Groups determine dot colour in all views (Today, Eisenhower, Tendrils, Focus)
- The `remove_ungrouped` trigger removes the ungrouped membership as soon as any real group is added

## `nodes`

The central table. A **node** is either a **task** (something to do) or an **event** (something happening at a specific time). Everything in the planner is a node.

```sql
CREATE TABLE IF NOT EXISTS nodes (
  id                         TEXT PRIMARY KEY,
  project_id                 TEXT,
  arc_id                     TEXT,
  title                      TEXT NOT NULL,
  node_type                  TEXT NOT NULL DEFAULT 'task'
                                 CHECK(node_type IN('task','event')),
  planned_start_at           DATETIME,
  due_at                     DATETIME,
  actual_completed_at        DATETIME,
  estimated_duration_minutes INTEGER,
  importance_level           INTEGER NOT NULL DEFAULT 0
                                 CHECK(importance_level BETWEEN 0 AND 4),
  computed_urgency_level     INTEGER NOT NULL DEFAULT 0
                                 CHECK(computed_urgency_level BETWEEN 0 AND 4),
  is_completed               BOOLEAN DEFAULT 0,
  is_locked                  BOOLEAN DEFAULT 0,
  is_overdue                 BOOLEAN DEFAULT 0,
  is_recovery                BOOLEAN DEFAULT 0,
  is_pinned                  BOOLEAN DEFAULT 0,
  recovery_set_at            TIMESTAMP,
  created_at                 TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at                 TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  recurrence_rule            TEXT,
  recurrence_exceptions      TEXT,
  is_frog_pinned             BOOLEAN DEFAULT 0,
  is_pre_node                BOOLEAN DEFAULT 0,
  tendril_pos_x              REAL,
  tendril_pos_y              REAL,
  is_routine                 INTEGER DEFAULT 0,
  routine_id                 TEXT,
  FOREIGN KEY(project_id) REFERENCES projects(id) ON DELETE SET NULL,
  FOREIGN KEY(arc_id)     REFERENCES arcs(id)     ON DELETE SET NULL
);
```

| Column | Type | Description |
|---|---|---|
| `id` | TEXT | UUID |
| `project_id` | TEXT | FK to `projects.id`. NULL if not linked to a project |
| `arc_id` | TEXT | FK to `arcs.id`. NULL if not linked to an arc. Can be set independently of `project_id` |
| `title` | TEXT | The task or event name shown in all views |
| `node_type` | TEXT | `'task'` or `'event'`. Events render as hollow circles; tasks render as filled dots |
| `planned_start_at` | DATETIME | When the user intends to work on this (schedule date). Drives "Today" section inclusion. Can be date-only (`YYYY-MM-DD`) or full datetime (`YYYY-MM-DDTHH:MM:SS`) |
| `due_at` | DATETIME | Hard deadline. Drives "Overdue" section and urgency scoring. Separate from `planned_start_at` |
| `actual_completed_at` | DATETIME | Set to the current UTC timestamp when `completeNode()` is called |
| `estimated_duration_minutes` | INTEGER | How long the task is expected to take. Drives dot size in all views |
| `importance_level` | INTEGER | `0` = normal, `1` = important. Used in Eisenhower quadrant placement and urgency scoring. Range 0â€“4 (currently only 0 and 1 are used) |
| `computed_urgency_level` | INTEGER | Calculated value 0â€“4 from `computeUrgencyLevel()` â€” combines `importance_level`, `due_at`, and current time. Recomputed on every load in `hydrateRows()` |
| `is_completed` | BOOLEAN | `1` when done. Completed nodes are excluded from `loadNodes()` but included in `loadAllNodes()` |
| `is_locked` | BOOLEAN | `1` = node cannot be dragged in TendrilsView. Set manually |
| `is_overdue` | BOOLEAN | Recomputed fresh on every load in `hydrateRows()` â€” do not trust the stored value as the source of truth |
| `is_recovery` | BOOLEAN | `1` if the node was rescheduled after its deadline passed. Set by `rescheduleNode()` |
| `is_pinned` | BOOLEAN | `1` = pinned to top of Today view task list |
| `recovery_set_at` | TIMESTAMP | When `is_recovery` was set to `1` â€” used for display in `TaskDetailPanel` |
| `created_at` | TIMESTAMP | Auto-set on insert |
| `updated_at` | TIMESTAMP | Auto-updated by the `nodes_ts` trigger on every `UPDATE` |
| `recurrence_rule` | TEXT | iCal-style RRULE string (e.g. `FREQ=WEEKLY;BYDAY=MO,WE,FR`). Stored on recurring event template nodes only; per-instance nodes have this set to `NULL` |
| `recurrence_exceptions` | TEXT | JSON array of date strings (e.g. `["2026-03-15","2026-03-22"]`). Dates where this recurring event was skipped or already handled |
| `is_frog_pinned` | BOOLEAN | `1` = this is the user's designated "frog" (the hardest most important task). Shown with special treatment in TodayView |
| `is_pre_node` | BOOLEAN | Reserved for "pre-node" workflow (blocking prerequisite concept). Not fully implemented in views yet |
| `tendril_pos_x` | REAL | Saved X position in the Tendrils canvas (pixels). NULL until the user has moved the node |
| `tendril_pos_y` | REAL | Saved Y position in the Tendrils canvas (pixels). NULL until the user has moved the node |
| `is_routine` | INTEGER | `1` if this node was spawned from a routine. Used in UI to show "edit in routines view" instead of the edit button |
| `routine_id` | TEXT | FK (soft, no constraint) to `routines.id`. Links a spawned node back to its parent routine template |

**Indexes:** `project_id`, `arc_id`, `due_at`, `planned_start_at`, `is_completed`, `is_overdue`, `routine_id`, `is_routine`

**Triggers:**
- `nodes_ts` â€” updates `updated_at` on every `UPDATE`
- `nodes_auto_ungrouped` â€” adds `g-ungrouped` to `node_groups` on every `INSERT`

**App usage:**
- `loadNodes()` â€” loads all incomplete nodes, hydrates them (compute `is_overdue`, `is_missed_schedule`, urgency, join groups)
- `loadAllNodes()` â€” same but includes completed nodes (used in Tendrils history)
- Dot size in all views is derived from `estimated_duration_minutes`
- Dot colour in all views is derived from the node's group colours (or arc colour)
- `TodayView` partitions nodes into Overdue / Missed Schedule / Today / Events / Suggestions sections based on `due_at`, `planned_start_at`, `is_completed`, and current date
- `EisenhowerView` places nodes in Q1â€“Q4 based on `importance_level` and `computed_urgency_level`
- `TendrilsView` uses `tendril_pos_x`/`tendril_pos_y` for canvas layout
- WTDI auto-advance: if an assignment (`due_at` set, `planned_start_at` in past) is not yet overdue, `hydrateRows()` pushes `planned_start_at` forward to today automatically

## `node_groups`

Junction table. Records which groups a node belongs to. One node can appear in multiple groups.

```sql
CREATE TABLE IF NOT EXISTS node_groups (
  node_id  TEXT NOT NULL,
  group_id TEXT NOT NULL,
  PRIMARY KEY(node_id, group_id),
  FOREIGN KEY(node_id)  REFERENCES nodes(id)          ON DELETE CASCADE,
  FOREIGN KEY(group_id) REFERENCES planner_groups(id) ON DELETE CASCADE
);
```

| Column | Type | Description |
|---|---|---|
| `node_id` | TEXT | FK to `nodes.id`. Cascades delete |
| `group_id` | TEXT | FK to `planner_groups.id`. Cascades delete |

**Triggers:**
- `remove_ungrouped` â€” when a real group is added, this trigger removes the `g-ungrouped` membership
- `readd_ungrouped_if_empty` â€” when the last real group is removed, this trigger restores `g-ungrouped`

**App usage:**
- Read via `getNodeGroups(nodeId)` on every node load â€” the `groups` array on `PlannerNode` is populated from this table
- Written by `addNodeGroup()`, `removeNodeGroup()`, `replaceNodeGroups()` in `plannerDb.ts`
- Managed in the task form's group multi-select

## `sub_tasks`

Checklist items nested inside a node. Displayed as a progress ring around the dot and a badge on hover.

```sql
CREATE TABLE IF NOT EXISTS sub_tasks (
  id           TEXT PRIMARY KEY,
  node_id      TEXT NOT NULL,
  title        TEXT NOT NULL,
  is_completed BOOLEAN DEFAULT 0,
  sort_order   INTEGER DEFAULT 0,
  created_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY(node_id) REFERENCES nodes(id) ON DELETE CASCADE
);
```

| Column | Type | Description |
|---|---|---|
| `id` | TEXT | UUID |
| `node_id` | TEXT | FK to `nodes.id`. Cascades delete â€” sub-tasks are destroyed with their parent node |
| `title` | TEXT | The checklist item text |
| `is_completed` | BOOLEAN | `0` or `1` |
| `sort_order` | INTEGER | Controls display order within the checklist |
| `created_at` | TIMESTAMP | Auto-set on insert |

**App usage:**
- `NODE_SELECT` counts `sub_total` and `sub_done` via subqueries on every node load
- The SVG ring on `DotNode` is drawn from `sub_total` / `sub_done`
- A `{done}/{total}` badge appears on the dot on hover
- Managed in the task detail / edit form

```sql
);
```

| Column | Type | Description |
|---|---|---|
| `id` | TEXT | UUID |
| `node_id` | TEXT | FK to `nodes.id`. Set to NULL if the node is later deleted (so the log entry is preserved) |
| `completed_at` | TIMESTAMP | When the node was marked complete |
| `duration_actual` | INTEGER | Minutes actually spent (currently not written â€” reserved for future stopwatch feature) |

**App usage:**
- A row is inserted by `completeNode()` in `plannerDb.ts` every time a task is marked done
- Not yet read back by any view â€” stored for future stats/analytics screens

## `user_capacity`

Single-row settings table for the user's work capacity. Always has exactly one row with `id = 'default'`, seeded at startup via `INSERT OR IGNORE`.

```sql
CREATE TABLE IF NOT EXISTS user_capacity (
  id            TEXT PRIMARY KEY DEFAULT 'default',
  daily_minutes INTEGER DEFAULT 480,
  peak_start    TEXT DEFAULT '09:00',
  peak_end      TEXT DEFAULT '12:00',
  updated_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

| Column | Type | Description |
|---|---|---|
| `id` | TEXT | Always `'default'` â€” only one row ever exists |
| `daily_minutes` | INTEGER | Total available work minutes per day. Default 480 (8 hours). Used for capacity % calculations in TodayView |
| `peak_start` | TEXT | Start of peak focus window (`HH:MM`). Default `'09:00'` |
| `peak_end` | TEXT | End of peak focus window (`HH:MM`). Default `'12:00'` |
| `updated_at` | TIMESTAMP | Last time this row was changed |

**App usage:**
- Loaded by `loadUserCapacity()` â†’ stored in `usePlannerStore` as `capacity`
- `daily_minutes` is used in TodayView to calculate what percentage of the day is filled by today's scheduled tasks
- `peak_start` / `peak_end` are reserved for future scheduling intelligence (not yet used in views)

## `tendril_edges`

Directed edges between nodes in the **TendrilsView** canvas. Lets the user draw links showing how tasks relate or depend on each other within a project.

```sql
CREATE TABLE IF NOT EXISTS tendril_edges (
  id         TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  source_id  TEXT NOT NULL,
  target_id  TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY(source_id) REFERENCES nodes(id) ON DELETE CASCADE,
  FOREIGN KEY(target_id) REFERENCES nodes(id) ON DELETE CASCADE
);
```

| Column | Type | Description |
|---|---|---|
| `id` | TEXT | UUID |
| `project_id` | TEXT | Which project this edge belongs to (used for filtering the canvas) |
| `source_id` | TEXT | The "from" node. Cascades delete |
| `target_id` | TEXT | The "to" node. Cascades delete |
| `created_at` | TIMESTAMP | Auto-set on insert |

**App usage:**
- Edges are drawn by the user in TendrilsView by dragging between nodes
- Rendered as SVG lines connecting dots on the canvas
- When either connected node is deleted, the edge is automatically removed via `ON DELETE CASCADE`

## `routines`

A **routine** is a template for a recurring task or event. It does not appear in views directly â€” instead, the app generates real `nodes` from it based on its schedule rules. The generated nodes have `is_routine = 1` and `routine_id` pointing back here.

```sql
CREATE TABLE IF NOT EXISTS routines (
  id                         TEXT PRIMARY KEY,
  title                      TEXT NOT NULL,
  description                TEXT,
  node_type                  TEXT NOT NULL DEFAULT 'task'
                                 CHECK(node_type IN('task','event')),
  recurrence_type            TEXT NOT NULL DEFAULT 'fixed'
                                 CHECK(recurrence_type IN('fixed','flexible','manual')),
  recurrence_rule            TEXT,
  period_unit                TEXT,
  period_count               INTEGER,
  arc_id                     TEXT,
  project_id                 TEXT,
  estimated_duration_minutes INTEGER,
  importance_level           INTEGER NOT NULL DEFAULT 0,
  active_from                DATE,
  active_until               DATE,
  is_archived                BOOLEAN DEFAULT 0,
  created_at                 TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at                 TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY(arc_id)     REFERENCES arcs(id)     ON DELETE SET NULL,
  FOREIGN KEY(project_id) REFERENCES projects(id) ON DELETE SET NULL
);
```

| Column | Type | Description |
|---|---|---|
| `id` | TEXT | UUID |
| `title` | TEXT | The task/event name used when spawning nodes |
| `description` | TEXT | Optional notes about the routine (not shown on spawned nodes) |
| `node_type` | TEXT | `'task'` or `'event'` â€” inherited by spawned nodes |
| `recurrence_type` | TEXT | `'fixed'` = specific days/times, `'flexible'` = a cadence without fixed days, `'manual'` = user triggers spawning manually |
| `recurrence_rule` | TEXT | iCal RRULE string for `'fixed'` routines |
| `period_unit` | TEXT | `'day'`, `'week'`, `'month'` â€” used for `'flexible'` recurrence cadence |
| `period_count` | INTEGER | How many `period_unit`s between occurrences for `'flexible'` routines |
| `arc_id` | TEXT | FK to `arcs.id`. Inherited by spawned nodes |
| `project_id` | TEXT | FK to `projects.id`. Inherited by spawned nodes |
| `estimated_duration_minutes` | INTEGER | Inherited by spawned nodes â€” drives dot size |
| `importance_level` | INTEGER | `0` or `1`. Inherited by spawned nodes |
| `active_from` | DATE | Date from which this routine starts spawning (`YYYY-MM-DD`). NULL = always active |
| `active_until` | DATE | Date after which this routine stops spawning. NULL = no end |
| `is_archived` | BOOLEAN | `1` = routine is hidden from the RoutinesView and no longer spawns nodes |
| `created_at` | TIMESTAMP | Auto-set on insert |
| `updated_at` | TIMESTAMP | Auto-updated by the `routines_ts` trigger |

**App usage:**
- Managed in `RoutinesView` â†’ `RoutineForm.tsx` â†’ `routineDb.ts`
- When a routine is saved/activated, the app calls `spawnRoutineOccurrences()` which generates real `nodes` rows and inserts them with `is_routine = 1` and `routine_id` set
- In TodayView and other views, routine-spawned nodes show "edit in routines view" instead of the normal edit/delete buttons
- Archiving sets `is_archived = 1` â€” the routine is hidden but its previously spawned nodes remain

## `routine_rules`

Each routine can have one or more **rules** â€” specific schedule configurations. For example, a routine could run Mon/Wed/Fri at 07:00 via one rule and also Saturday at 09:00 via a second rule.

```sql
CREATE TABLE IF NOT EXISTS routine_rules (
  id               TEXT PRIMARY KEY,
  routine_id       TEXT NOT NULL,
  sort_order       INTEGER DEFAULT 0,
  freq             TEXT NOT NULL DEFAULT 'weekly'
                       CHECK(freq IN('daily','weekly','monthly')),
  repeat_interval  INTEGER NOT NULL DEFAULT 1,
  days             TEXT,
  start_date       TEXT NOT NULL,
  end_mode         TEXT NOT NULL DEFAULT 'count'
                       CHECK(end_mode IN('count','date')),
  end_count        INTEGER,
  end_date         TEXT,
  start_time       TEXT,
  duration_minutes INTEGER,
  FOREIGN KEY(routine_id) REFERENCES routines(id) ON DELETE CASCADE
);
```

| Column | Type | Description |
|---|---|---|
| `id` | TEXT | UUID |
| `routine_id` | TEXT | FK to `routines.id`. Cascades delete â€” rules are destroyed with the routine |
| `sort_order` | INTEGER | Display order when showing multiple rules |
| `freq` | TEXT | `'daily'`, `'weekly'`, or `'monthly'` |
| `repeat_interval` | INTEGER | Every N units (e.g. `2` + `'weekly'` = every 2 weeks) |
| `days` | TEXT | JSON array of weekday numbers `[0â€“6]` (0 = Sunday). Used for weekly recurrence (e.g. `[1,3,5]` = Mon/Wed/Fri) |
| `start_date` | TEXT | `YYYY-MM-DD` â€” the first date this rule starts generating occurrences from |
| `end_mode` | TEXT | `'count'` = stop after N occurrences, `'date'` = stop on a specific date |
| `end_count` | INTEGER | Number of occurrences before stopping. Used when `end_mode = 'count'` |
| `end_date` | TEXT | `YYYY-MM-DD` stop date. Used when `end_mode = 'date'` |
| `start_time` | TEXT | `HH:MM` time of day for events. NULL for tasks |
| `duration_minutes` | INTEGER | Override for this rule's duration (if different from the routine's default) |

**App usage:**
- Read via `loadRoutineRules(routineId)` â€” every loaded `Routine` object has a `rules` array populated from this table
- The `generateOccurrenceDates()` function in `recurrence.ts` takes a rule and produces the list of date strings to spawn nodes on
- Managed in the `RoutineForm` as a list of schedule blocks the user can add/remove

## `routine_groups`

Junction table. Records which groups a routine belongs to. Spawned nodes inherit these group memberships.

```sql
CREATE TABLE IF NOT EXISTS routine_groups (
  routine_id TEXT NOT NULL,
  group_id   TEXT NOT NULL,
  PRIMARY KEY (routine_id, group_id),
  FOREIGN KEY (routine_id) REFERENCES routines(id)        ON DELETE CASCADE,
  FOREIGN KEY (group_id)   REFERENCES planner_groups(id)  ON DELETE CASCADE
);
```

| Column | Type | Description |
|---|---|---|
| `routine_id` | TEXT | FK to `routines.id`. Cascades delete |
| `group_id` | TEXT | FK to `planner_groups.id`. Cascades delete |

**App usage:**
- Read in `RoutineForm` to show which groups the routine is tagged with
- When nodes are spawned from a routine, `group_ids` from this table are passed to `insertSingleNode()` so the spawned nodes carry the same group memberships

## Cascade & Trigger Summary

| Event | Effect |
|---|---|
| Delete arc | `nodes.arc_id â†’ NULL`, `projects.arc_id â†’ NULL` |
| Delete project | `nodes.project_id â†’ NULL` |
| Delete group | All `node_groups` rows referencing it are deleted via CASCADE |
| Delete node | All `node_groups`, `sub_tasks`, `tendril_edges` rows for it are deleted via CASCADE |
| Delete routine | All `routine_rules`, `routine_groups` rows for it are deleted via CASCADE |
| Insert node | `nodes_auto_ungrouped` trigger adds `g-ungrouped` to `node_groups` |
| Add real group to node | `remove_ungrouped` trigger removes `g-ungrouped` from `node_groups` |
| Remove last real group from node | `readd_ungrouped_if_empty` trigger adds `g-ungrouped` back |
| Update node | `nodes_ts` trigger updates `updated_at` |
| Update routine | `routines_ts` trigger updates `updated_at` |
| Complete node | `completeNode()` sets `is_completed = 1`, `actual_completed_at = now`, inserts a `productivity_logs` row |
